Product & Technical Overview

KnightVision

AI-powered chess game analysis — Stockfish accuracy scores, LLM coaching commentary, and a real-time review board in the browser.
♟ Chess Analysis Platform ⚡ Real-time WebSocket Streaming 🤖 Multi-LLM Provider 📦 Docker Single-Command

1 What is KnightVision?

KnightVision is a full-stack chess game analysis tool that combines the world-class Stockfish chess engine with AI language models to deliver move-by-move coaching commentary. A player pastes a PGN game, clicks Analyse, and within seconds sees every move colour-coded by quality — Brilliant, Best, Inaccuracy, Blunder — alongside a coaching explanation, an evaluation bar, and the top alternative moves the engine recommended.

The goal is to make post-game analysis as accessible as a chess.com or Lichess review, but with the flexibility to run entirely locally (no cloud required) or swap in a free cloud LLM (Groq) when needed.

Core value proposition: Upload any PGN → get a fully annotated game with accuracy scores, move-quality badges, and plain-English coaching comments — in under 60 seconds — with zero setup beyond docker compose up.

2 Key Features

♟

Stockfish Move Analysis

Every position evaluated at depth 18. Centipawn loss computed per move from the moving player's perspective. Top 3 alternatives returned for each position.

🤖

AI Coaching Commentary

LLM generates a 1–2 sentence plain-English explanation for every move. Supports Ollama (local), Groq, and HuggingFace. Falls back to templates if LLM times out.

🎨

Colour-Coded Move List

All 7 move categories displayed with distinct colours and badges. Move list is scrollable with auto-scroll to the current move. WCAG 2.1 AA contrast verified.

📊

Accuracy Scores

White and Black accuracy percentages calculated across the full game. Displayed in the game header with colour-coded indicators (green ≥80%, amber 60–79%, red <60%).

⚡

Real-Time Streaming

Results stream via WebSocket as each move is analysed — no waiting for the whole game. Late-joining clients receive buffered replay instantly without animation delays.

📱

Responsive & Accessible

Fully responsive layout — desktop two-column, mobile bottom-sheet with swipe navigation. Keyboard nav (arrow keys, Space, Home, End). WCAG 2.1 AA compliant.

🔄

Runtime LLM Switching

Switch between Ollama, Groq, and HuggingFace at runtime via the UI dropdown — no server restart required. Provider health displayed live.

🐳

Single-Command Docker

Three services (backend, frontend, Ollama) start with one command. Ollama model is auto-pulled and cached in a Docker volume on first run.

3 Move Classification System

Every move is classified into one of 7 quality tiers based on the centipawn loss — the difference between the engine's evaluation before and after the move, from the moving player's perspective.

Badgecp_loss thresholdMeaning
Brilliant= 0 + sacrificeZero loss + piece sacrifice detected
Great≤ 5Near-perfect move
Best≤ 10Engine's top recommendation
Good≤ 20Solid, slightly sub-optimal
Inaccuracy≤ 50Noticeable error
Mistake≤ 150Significant error
Blunder> 150Game-changing error

Brilliant detection: A move must have cp_loss = 0 AND involve a piece sacrifice (giving up material that isn't immediately recoverable). This prevents trivially recapturing moves from being labelled Brilliant.

Configurable: All thresholds are adjustable in config.yaml under classification.thresholds — no code change needed.

4 LLM Provider System

KnightVision's LLM layer is built around a pluggable provider registry with automatic fallback. All providers implement the same interface (generate(prompt) → str + check_health() → bool), making them interchangeable at runtime.

ProviderModelKey RequiredBest For
Ollama (default) llama3.1:8b None — runs locally Privacy, offline use, no API costs
Groq llama3-8b-8192 GROQ_API_KEY (free tier) Fastest inference, hosted deployments
HuggingFace Mistral-7B-Instruct-v0.2 HUGGINGFACE_API_KEY (free) Alternative cloud option
Fallback Template engine None — always available Guaranteed comment if all LLMs fail

Fallback Chain

Every comment request goes through generate_with_fallback(): it tries the active provider, and if that fails (network error, timeout, no API key), automatically tries the next registered provider, finally falling back to a template comment. Analysis never fails due to an LLM being unavailable.

10-Second Timeout

Each LLM call has a 10-second timeout. If the model doesn't respond in time, the fallback kicks in immediately — preventing slow LLM responses from blocking the entire analysis pipeline.

9 Streaming & Resilience

📡

WebSocket Streaming

Results are pushed to the browser as each move is processed — the board updates in real time. A 10-second heartbeat keeps the connection alive through proxies and firewalls.

⏪

Buffered Replay

A client that reconnects mid-analysis immediately receives all prior moves with buffered: true — rendered instantly without animation — then seamlessly transitions to live streaming.

⏱

Stall Detection

The frontend monitors time since last WebSocket message. If no message arrives in 30 seconds while analysis is active, status transitions to stalled and a warning banner is shown.

🛡

Per-Move Error Isolation

If a single move fails (Stockfish error, LLM timeout), the error is isolated to that move — a fallback "Good" category and template comment are used — and analysis continues for all remaining moves.

11 Deployment

Local Docker (Default)

git clone <repo> && cd knightvision
cp .env.example .env
docker compose up --build

Three services start: backend (port 8000), frontend (port 3000), Ollama (port 11434). The llama3.1:8b model (~4 GB) is pulled automatically on first run and cached in a Docker volume.

Cloud (Railway + Groq)

railway init
railway up
# set in Railway dashboard:
# GROQ_API_KEY=...
# LLM_DEFAULT_PROVIDER=groq

Backend deploys from backend/Dockerfile. Frontend deploys to Vercel/Netlify with VITE_API_BASE_URL pointing to the Railway backend URL.

Prod Compose Override (no Ollama)

docker compose -f docker-compose.yml -f docker-compose.prod.yml up --build

Removes the Ollama sidecar and switches the default provider to Groq. Useful for low-memory machines or when a cloud LLM is preferred locally.
