🔄 Request / Data Flow

1

User uploads PGN

Frontend reads file / paste → validates with chess.js → POST /api/games/upload → backend returns game_id (UUID4) + metadata + total_moves (202 Accepted) → Frontend immediately opens WebSocket ws://.../ws/games/{game_id}/analysis

2

Backend parses PGN + starts background analysis

python-chess extracts all moves, FEN per ply, PGN tag metadata → FastAPI BackgroundTask launches analysis pipeline → MemoryStore stores GameAnalysis object (status: ANALYSING) → WebSocket sends initial progress message

3

Stockfish evaluates each position

For each ply: send FEN via UCI → get score_cp, best_move_uci, pv, top 3 candidates → compute cp_loss = max(0, eval_before − eval_after) from mover's perspective · depth 18 default · ~50–200ms per position

4

Opening Detector + Move Classifier

ECO JSON lookup on first N half-moves → attach opening_name + opening_eco if applicable · Move Classifier maps cp_loss → category (Brilliant / Great / Best / Good / Inaccuracy / Mistake / Blunder) using config.yaml thresholds · Brilliant requires additional sacrifice/tactic check

5

LLM generates commentary (with fallback)

ProviderRegistry.generate_with_fallback() → asyncio.wait_for(active_provider.generate(prompt), timeout=10s) → if LLM responds: comment_source="llm" · if timeout/error: template fallback comment, comment_source="fallback" · analysis never blocked by LLM failure

6

WebSocket streams results to frontend

Each completed MoveAnalysis is pushed as move_result message → frontend updates move list, board, eval bar incrementally → on all moves done: complete message with GameSummary (accuracy scores, category counts, turning point) → progress bar fills to 100%

WebSocket Message Types

progress

Sent every 5 moves as keep-alive / UI progress indicator

  • game_id, moves_done, total_moves
  • status: "analysing"

move_result

Sent for each completed move — contains full MoveAnalysis object

  • move_index, total_moves
  • move: {san, uci, fen_before/after, eval_before/after_cp, cp_loss, best_move, top_candidates, category, category_symbol, comment, comment_source, opening_name?, opening_eco?}

complete

Sent once all moves are done — includes full GameSummary

  • summary: {white_accuracy, black_accuracy, white/black_category_counts, total_moves, turning_point_move_index}

error

Non-fatal — analysis continues. Failed move gets "Good" + fallback comment.

  • message: error description
  • move_index: which move failed (optional)

🏷️ Move Classification System (7-Tier)

!! Brilliant

cp_loss: 0 (only best move) + sacrifice/tactic

Engine's top move AND involves material sacrifice or deep combination. Requires cp_loss == 0 + sacrifice detection. Rare.

! Great

cp_loss: 0–5

Excellent move, significantly stronger than alternatives. Not the only best move.

✓ Best

cp_loss: 0–10

Engine's top choice. Objectively optimal or near-optimal. Colour: #a78bfa (purple)

~ Good

cp_loss: 10–20

Solid, reasonable move. Keeps advantage or limits damage. Colour: #22d3ee (cyan)

?! Inaccuracy

cp_loss: 20–50

Slightly suboptimal. Minor positional cost. Colour: #fbbf24 (yellow)

? Mistake

cp_loss: 50–150

Clear error giving opponent meaningful advantage. Colour: #fb923c (orange)

?? Blunder

cp_loss: 150+

Severe error. Significant material or positional loss. Colour: #f87171 (red)

⚙ All thresholds stored in config.yaml → classifier.thresholds — adjustable without code changes. Formula: cp_loss = max(0, eval_before − eval_after) where both evals are from the perspective of the player who just moved.

🎯 Accuracy Score (Phase 1 — Confirmed)

accuracy_percent = 100 − (average_cp_loss / MAX_CP_LOSS_FOR_SCALE × 100) MAX_CP_LOSS_FOR_SCALE = 300 # calibrated to match Chess.com scale approximately result = clamp(accuracy_percent, 0, 100)

Computed separately for White and Black after full analysis. Zero additional Stockfish calls — reuses cp_loss values already collected. Adds <1ms to analysis time. Displayed in GameHeader as "White: 61.2% · Black: 78.4%".

📝 Key Architectural Decisions

DecisionRationale
Pre-compute on upload, not on-demand All Stockfish + LLM analysis runs as a background task immediately after upload. Zero lag during interactive playback. Results streamed via WebSocket as computed.
WebSocket for streaming, REST for cache WebSocket pushes each MoveAnalysis as it's ready (real-time UX). REST GET for fetching full cached analysis after completion. Polling is explicitly avoided — race conditions during playback.
LLM Adapter Pattern (BaseLLMProvider ABC) Single abstract interface: generate(prompt) → str + health_check() → bool. Swapping providers = change one config value. ProviderRegistry handles timeout + fallback transparently.
LLM is optional — always fallback-safe asyncio.wait_for(10s timeout) around every LLM call. On timeout or error → template fallback comment. App fully functional without any LLM. comment_source field tells frontend which was used.
Stockfish as long-lived subprocess Single Stockfish process managed by python-chess for the analysis session. NOT a new subprocess per move — avoids startup overhead and binary re-download.
In-memory store (no DB) Python dict keyed by UUID game_id with a threading.Lock for thread safety. append_move() is thread-safe for the background analysis task. Designed for clean DB migration in Phase 2.
Centipawn loss from mover's perspective eval_before_cp and eval_after_cp are always from the perspective of whoever just moved. cp_loss = max(0, eval_before − eval_after). This normalizes the sign — positive cp_loss is always a deterioration for the mover.
Opening detection: client-side ECO JSON Bundled ~200KB ECO JSON, no external API. Match first N half-moves against ECO database. Return deepest match. Graceful fallback to "Unknown Opening".
